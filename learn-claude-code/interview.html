简单
什么是 Claude Code？它和普通 IDE 插件有什么区别？
基础概念
▼
参考答案

Claude Code 是 Anthropic 推出的 AI 编程助手 CLI 工具，它不是简单的代码补全插件，而是一个能理解项目上下文、执行复杂任务的智能开发伙伴。

与普通 IDE 插件的区别：

  • 交互方式：自然语言对话 vs 代码补全提示
  • 上下文理解：完整项目级别 vs 当前文件
  • 执行能力：可执行 Bash、Git 命令 vs 只提供建议
  • 任务复杂度：可多步骤自主执行 vs 单次补全

🎯 面试重点

  • 强调 Tool Use 架构，不只是聊天
  • 能举出具体例子（如：重构多文件 vs 单行补全）
  • 理解 Agent 模式的概念
简单
Claude Code 的核心工作原理是什么？
基础概念
▼
参考答案

Claude Code 基于工具调用（Tool Use）架构工作：

  1. 用户用自然语言描述需求
  2. Claude 分析任务，选择并调用合适的工具（Read、Write、Bash 等）
  3. 工具执行后返回结果
  4. Claude 根据结果决定下一步（继续调用工具或回复用户）

这是一个循环过程，直到任务完成。

🎯 面试重点

  • 必须提到 "Tool Use" 或 "工具调用"
  • 能画出循环流程图更佳
  • 理解 AI 是如何通过工具与环境交互的

Claude Code 有哪些核心工具？各自的使用场景是什么？
工具系统
▼
参考答案

文件操作类：

  • FileRead - 读取文件内容，支持指定行范围
  • FileWrite - 创建新文件或覆盖已有文件
  • FileEdit - 精确修改文件中的特定内容

代码搜索类：

  • Grep - 在代码中搜索文本模式，支持正则
  • Glob - 按模式查找文件，如 **/*.tsx

命令执行类：

  • Bash - 执行 shell 命令（git、npm 等）

智能工具类：

  • Agent - 委派子任务给专门的 Agent
  • WebSearch / WebFetch - 搜索网络信息

🎯 面试重点

  • 能分类列举（文件/搜索/命令/智能）
  • 理解 FileEdit 和 FileWrite 的区别
  • 知道 Bash 工具的特殊性（需要权限控制）
困难
从源码角度，Claude Code 的工具系统是如何设计的？
工具系统 / 底层原理
▼
参考答案

工具系统采用分层架构设计：

  1. 基类层（Tool.ts）：定义统一接口，包括 name、description、parameters（Zod Schema）、execute 方法
  2. 实现层（tools/*.tsx）：45+ 个具体工具实现，如 BashTool、FileReadTool
  3. 注册层（tools.ts）：工具注册表，管理所有可用工具
  4. 权限层（hooks/toolPermission）：检查工具调用是否被允许
  5. 调用层（QueryEngine）：AI 根据上下文选择工具，生成调用参数

关键设计：所有工具遵循相同模式，便于扩展和权限控制。危险工具（如 BashTool）有额外的安全模块。

🎯 面试重点

  • 能画出 5 层架构图
  • 理解 Tool 基类的核心属性
  • 知道 Zod Schema 用于参数验证
  • 提到 BashTool 的特殊安全处理

Claude Code 的权限系统是如何工作的？
权限安全
▼
参考答案

权限系统采用多层安全检查机制：

Layer 1 - 配置规则：检查 settings.json 中的 allow/deny 规则，支持通配符匹配

Layer 2 - 智能分类器：如果配置未匹配，使用 bashClassifier 分析命令语义，给出 allow/ask/deny 判断

Layer 3 - AST 分析：对复杂命令进行语法树分析，检测隐藏的管道操作等

三种权限模式：

  • Manual - 每次询问（最安全）
  • Auto - AI 自动判断（推荐）
  • Full Auto - 极少询问（仅隔离环境）

🎯 面试重点

  • 必须提到 3 层检查
  • 理解分类器的作用（允许/询问/拒绝）
  • 能比较 3 种权限模式的适用场景
困难
如何设计一个团队级的 Claude Code 安全配置？
权限安全 / 场景题
▼
参考答案

1. 全局配置（~/.claude/settings.json）：

{
  "permissions": {
    "defaultMode": "auto",
    "allow": [
      "Bash:git status",
      "Bash:git diff",
      "Bash:pytest*",
      "Bash:npm run*"
    ],
    "deny": [
      "Bash:rm -rf*",
      "Bash:git push --force*",
      "Write:.env*"
    ]
  }
}

2. 项目级配置（.claude/CLAUDE.md）：

  • 定义技术栈和代码规范
  • 说明项目的特殊约束
  • 记录常用的开发命令

3. Git 集成：

  • 将 .claude/ 加入版本控制，统一团队配置
  • 配置 pre-commit hook 防止敏感信息泄露

🎯 面试重点

  • 分层配置（全局 + 项目级）
  • 安全优先的 allow/deny 策略
  • 版本控制同步配置
  • 考虑新人上手体验

什么是 Agent 模式？与普通模式有什么区别？
Agent 模式
▼
参考答案

Agent 模式让 Claude 能够自主规划并执行多步骤任务，而不是每步都询问用户。

核心区别：

维度 普通模式 Agent 模式
交互方式 每步询问 自主执行
任务规划 单轮对话 自动拆解任务
适用场景 简单、单文件操作 复杂、多文件重构

🎯 面试重点

  • 强调 "自主规划" 和 "多步骤"
  • 能举例说明使用场景（重构 10 个文件 vs 修改 1 个文件）
  • 理解效率和安全性的权衡
困难
解释 Coordinator 模式的架构设计
Agent 模式 / 底层原理
▼
参考答案

Coordinator 模式是一种多 Agent 协调架构：

  • Coordinator（协调者）：主 Agent，负责任务拆解、委派、结果汇总
  • Workers（工作者）：子 Agent，执行具体的子任务

工作流程：

  1. Coordinator 接收复杂任务，拆解成子任务
  2. 通过 AgentTool 启动多个 Worker，并行执行
  3. Worker 通过 <task-notification> 异步返回结果
  4. Coordinator 汇总结果，向用户报告

Worker 的限制：只有基础工具（Bash/Read/Edit），不能直接对话，完成即退出

🎯 面试重点

  • 类比：项目经理 vs 团队成员
  • 异步通信机制（通知而非直接对话）
  • Worker 的工具限制设计意图
困难
场景：团队想引入 Claude Code，但担心安全风险，你会如何设计推广方案？
场景题 / 综合
▼
参考答案

1. 风险评估阶段（第 1 周）：

  • 梳理团队的工作流程和代码库敏感度
  • 识别高风险操作（生产环境、敏感配置）
  • 制定允许/禁止的工具清单

2. 试点阶段（第 2-3 周）：

  • 选择 2-3 名志愿者，限制在测试项目使用
  • 配置严格的 Manual 模式，记录所有操作
  • 每日复盘，收集问题和改进建议

3. 配置阶段（第 4 周）：

  • 基于试点经验，制定团队配置规范
  • 设置 deny 规则保护关键文件和命令
  • 编写团队专用的 CLAUDE.md 项目文档

4. 推广阶段（第 5-6 周）：

  • 组织培训，分享最佳实践
  • 建立内部支持渠道（答疑、问题反馈）
  • 定期审计使用情况，持续优化配置

🎯 面试重点

  • 分阶段推进，降低风险
  • 从试点到推广的策略
  • 安全与效率的平衡
  • 持续改进的思维
