Claude Code 是什么？

核心概念

Claude Code 不是一个简单的"聊天机器人"，而是一个基于工具调用（Tool Use）架构的 AI 编程环境。它的核心设计理念是：

  • 工具即能力 — 每个工具都是与外部世界交互的能力单元
  • AI 编排者 — Claude 负责理解意图、选择工具、编排执行流程
  • 人在回路 — 关键决策点保留人工确认（可配置）

底层原理 1

工具系统架构

理解 Tool 基类、工具注册机制、执行流程

🔧 工具的本质

在 Claude Code 中，工具（Tool）是 AI 与外部世界交互的唯一方式。每个工具都有：

  • 名称 — 唯一标识（如 Bash、FileRead、Grep）
  • 描述 — AI 用它来判断何时调用该工具
  • 参数 Schema — 定义需要什么输入（Zod 验证）
  • 执行函数 — 实际执行逻辑

🏗️ 工具系统的分层架构

1
工具基类 (Tool.ts)

定义统一接口：输入验证、执行、结果处理、权限检查

2
具体工具实现 (tools/*.tsx)

BashTool、FileReadTool、GrepTool 等 45+ 个工具

3
工具注册表 (tools.ts)

管理所有可用工具，负责工具发现和调用分发

4
权限控制层 (hooks/toolPermission)

检查工具调用是否被允许，决定是否询问用户

5
AI 调用层 (QueryEngine)

Claude 根据上下文选择合适工具，生成调用参数

📋 工具基类的设计

Tool 基类核心属性

📝

🔍 示例：BashTool 的内部结构

Bash 工具的特殊性

高风险工具

BashTool 是最复杂的工具之一，因为它可以执行任意命令，安全风险高。它的实现包含多个安全层：

BashTool 安全模块

tools/BashTool/
🛡️
bashPermissions.ts 权限规则匹配、分类器调用
🔒
bashSecurity.ts AST 语法分析、危险命令检测
📁
pathValidation.ts 路径约束检查
🏖️
shouldUseSandbox.ts 是否使用沙箱执行

💡 关键理解

为什么工具系统如此设计？
  • 统一接口 — 所有工具遵循相同模式，便于 AI 理解和使用
  • 可扩展 — 新增工具只需继承基类，自动获得权限控制
  • 可组合 — 工具可以嵌套（如 AgentTool 调用其他工具）
底层原理 2

权限系统原理

多级安全检查、分类器、语义分析

🛡️ 权限判断的完整流程

当你运行一个命令时，Claude Code 会进行多层安全检查，决定是否自动执行、询问确认、或拒绝执行。

用户请求执行命令
↓
Layer 1: 配置规则检查
settings.json allow/deny
↓ 匹配？
明确允许
或
明确拒绝
或
未匹配 → 继续
↓
Layer 2: 分类器检查
bashClassifier.ts
↓
allow / ask / deny
↓
Layer 3: 语义分析
AST 语法树分析
↓
最终决策
执行 / 询问 / 拒绝

🔍 Layer 1: 配置规则匹配

首先检查 settings.json 中的 allow 和 deny 规则。支持通配符匹配：

配置规则匹配逻辑 
// 规则匹配示例
allow: ["Bash:git status*"]  →  匹配 "git status"、"git status --short"
allow: ["Bash:*"]            →  匹配所有 Bash 命令
deny: ["Bash:rm*"]           →  拒绝所有删除命令

🤖 Layer 2: 智能分类器

Bash 命令分类器

AI 驱动

如果配置规则未匹配，进入分类器（bashClassifier.ts）。它会分析命令的语义，给出三种判断：

  • allow — 安全命令，自动执行（如 git status、ls）
  • ask — 不确定，询问用户（如 npm install）
  • deny — 危险命令，建议拒绝（如 rm -rf /）

分类器如何判断？

命令类型 判断依据 结果
ls、cat、grep 只读操作，无副作用 allow
npm install、pip install 修改环境，但可预期 ask
rm -rf /、mkfs 破坏性操作 deny
curl | sh 管道执行远程脚本 deny

🔬 Layer 3: 语义分析（AST）

对于复杂的复合命令，Claude Code 使用语法树分析（AST）来理解命令的真实意图：

AST 分析示例 
# 复合命令
git add . && git commit -m "update" && git push

# AST 分析结果：
├─ 命令1: git add .          → 修改暂存区
├─ 操作符: &&                → 顺序执行
├─ 命令2: git commit         → 创建提交
├─ 操作符: &&                → 顺序执行
└─ 命令3: git push           → 推送到远程（高风险）

📊 权限决策矩阵

场景 配置规则 分类器 最终结果
git status allow 列表匹配 无需判断 ✅ 自动执行
rm file.txt 无匹配 deny（删除） ❌ 拒绝执行
npm test 无匹配 ask（不确定） ❓ 询问用户
为什么需要多层检查？

单层检查容易被绕过。例如：用户配置 allow: ["cat *"]，但攻击者可能用 cat /etc/passwd | send_to_attacker。多层检查（配置 + 分类器 + AST）确保安全性。

底层原理 3

Agent 协调机制

多 Agent 并行、任务委派、上下文管理

🤖 Agent 模式的核心问题

复杂任务（如"重构整个项目"）需要：

  • 搜索所有相关文件
  • 并行修改多个文件
  • 运行测试验证
  • 汇总结果报告

如果单线程顺序执行，会非常慢。Agent 协调模式通过委派子任务给多个 Worker Agent 来解决这个问题。

🏗️ Coordinator 架构

Coordinator Agent
主协调者
↓ 委派任务
Worker 1
搜索代码
Worker 2
修改文件 A
Worker 3
修改文件 B
Worker 4
运行测试
↑ 返回结果
Coordinator
汇总结果 → 用户

📋 Coordinator 的职责

Coordinator 的工作

📋
任务拆解 将复杂任务拆成可并行的子任务
🚀
委派执行 通过 AgentTool 启动 Worker
⏳
等待结果 异步接收 Worker 完成通知
🔄
汇总反馈 整合结果，向用户报告

Worker 的限制

🛠️
工具限制 只有基础工具（Bash/Read/Edit）
💬
不能直接对话 结果通过通知发给 Coordinator
🎯
专注单一任务 完成即退出，不维护长期状态

🔄 Agent 通信机制

基于通知的异步通信

核心机制

Worker 不直接与用户对话，而是通过 <task-notification> XML 消息通知 Coordinator：

Worker 完成通知示例 
<task-notification>
  <task-id>worker-123</task-id>
  <status>completed</status>
  <summary>Found 5 files matching pattern</summary>
  <output-file>/path/to/result.json</output-file>
</task-notification>

🎯 何时使用 Agent 模式？

场景 普通模式 Agent 模式
修改单个文件 ✅ 适合 ❌ 过度设计
搜索代码 ✅ 适合 ❌ 过度设计
重构 10+ 个文件 ❌ 太慢 ✅ 并行处理
批量修复相似 Bug ❌ 重复工作 ✅ 每个文件一个 Worker
同时运行测试+检查 ❌ 顺序执行 ✅ 并行执行
理解 Coordinator 的本质

Coordinator 模式类似于项目管理：项目经理（Coordinator）将大项目拆成小任务，分配给团队成员（Workers），收集结果，向老板汇报。团队成员不直接向老板汇报。

底层原理 4

Hooks 事件系统

事件驱动架构、Hook 注册与执行

🪝 什么是 Hooks？

Hooks（钩子）是 Claude Code 的事件响应机制。当特定事件发生时（如文件保存、会话启动），自动执行你预定义的操作。

类比理解：

  • Git Hooks —— git 操作前后自动执行脚本
  • React Hooks —— 组件生命周期中执行副作用
  • Claude Code Hooks —— 工具使用前后自动执行操作

🎯 可用的 Hook 类型

生命周期 Hooks

🚀
SessionStart 会话启动时触发
💬
UserPromptSubmit 用户发送消息前

文件操作 Hooks

📝
FileWrite 写入文件后
✏️
FileEdit 编辑文件后

工具使用 Hooks

🔧
PreToolUse 工具使用前（可拦截）
✅
PostToolUse 工具使用后

⚙️ Hook 执行流程

事件触发

如：Claude 执行 FileWrite 工具保存文件

Matcher 匹配

检查文件的 matcher 规则（如 *.py）

Hook 执行

按顺序执行所有匹配的 hook

原操作继续

Hook 执行完毕，返回原流程

🔍 Hook 实现原理

Hook 注册与执行机制 
// 1. 注册 Hook
{
  "hooks": {
    "FileWrite": [
      {
        "matcher": "*.py",           // 匹配规则
        "hooks": [
          {
            "type": "command",       // Hook 类型
            "command": "black $CLAUDE_FILE"  // 执行命令
          }
        ]
      }
    ]
  }
}

// 2. 执行时环境变量
$CLAUDE_FILE    → 被修改的文件路径
$CLAUDE_EDIT    → 编辑内容（FileEdit 时）

💡 Hook 的应用场景

场景 Hook 配置 效果
自动格式化 FileWrite → prettier 保存时自动格式化
自动测试 FileEdit → pytest 修改后自动运行测试
TypeScript 检查 FileWrite → tsc 保存时类型检查
Git 提交前检查 PreToolUse (Bash:git commit) 提交前强制测试通过
Hook 的注意事项
  • Hook 阻塞执行 — 如果 Hook 很慢（如运行全部测试），会卡住主流程
  • Hook 失败会报错 — 如果 Hook 返回非零退出码，操作会被取消
  • 可以用 & 后台执行避免阻塞："command": "pytest &"
底层原理 5

查询引擎

理解 QueryEngine、消息流转、上下文管理

🔍 QueryEngine 是什么？

QueryEngine 是 Claude Code 的核心大脑。它负责：

  • 接收用户输入，构建上下文
  • 调用 Claude API，获取 AI 响应
  • 解析 AI 的工具调用请求
  • 执行工具，将结果返回给 AI
  • 管理对话历史和状态

🔄 完整请求处理流程

1. 用户输入
↓
2. 构建 Prompt
系统提示 + 上下文 + 用户消息
↓
3. 调用 Claude API
streaming 流式响应
↓
4. 解析响应
文本回复 vs 工具调用
A. 纯文本回复
直接展示给用户
→
结束
或
B. 工具调用
执行工具
→
返回结果给 AI
继续循环

📊 上下文管理

Token 限制与压缩

关键技术

Claude API 有上下文长度限制（如 200K tokens）。QueryEngine 需要智能管理上下文：

上下文组成

📋
系统提示 告诉 Claude 如何行事（固定）
💾
记忆 持久化的项目信息
💬
对话历史 用户和 Claude 的消息
🛠️
工具结果 文件内容、命令输出

压缩策略

🗜️
/compact 手动压缩历史对话
✂️
自动截断 丢弃最旧的消息
📝
智能摘要 保留关键信息

🎯 为什么需要循环？

AI 不能一次性完成复杂任务，因为：

  • 信息不足 — AI 不知道文件内容，需要读取后才能决定
  • 推理链 — 多步推理，每步依赖上一步结果
  • 错误恢复 — 如果工具执行失败，需要重新尝试
典型的多轮交互 
用户: 修复 login 函数的 bug

Claude: 🔧 调用 Grep 搜索 login 函数

[工具结果: 找到 3 个文件]

Claude: 🔧 调用 FileRead 读取 src/auth/login.ts

[工具结果: 文件内容...]

Claude: 🔧 调用 FileEdit 修复第 45 行的空指针问题

[工具结果: 修改成功]

Claude: ✅ 已修复！login 函数的第 45 行缺少 null 检查...
