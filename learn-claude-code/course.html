🔍 关键文件详解

main.tsx - 应用的入口点

📍 位置

restored-src/src/main.tsx
约 80 万行，是整个应用的核心

🗣️ 人话解释

这就像是餐厅的总控台。当你运行 claude 命令时，代码首先进入这里。main.tsx 负责：

  • 解析命令行参数（你要用哪个模型、哪个权限模式）
  • 初始化所有模块（工具系统、命令系统、权限系统）
  • 启动用户界面（Ink 终端 UI）
  • 进入主循环，等待用户输入

💡 为什么这样设计？

把所有逻辑放在一个文件看似混乱，其实这是有意为之的架构选择：

  • CLI 工具的生命周期是单一的（启动 → 运行 → 退出）
  • 集中管理避免了模块间的循环依赖问题
  • 便于打包和发布（npm 包只有一个入口）
Tool.ts - 工具基类定义

📍 位置

restored-src/src/Tool.ts
约 1200 行，定义所有工具的通用接口

这是整个工具系统的宪法文件。所有工具（Bash、FileRead、Grep 等）都必须遵守这里定义的规范。

Tool 基类的核心属性（简化版）
export type ToolDef = {
  name: string;           // 工具名称，如 "Bash"
  description: string;    // 功能描述，AI 用此理解工具用途
  parameters: z.ZodSchema; // Zod Schema，定义参数结构
  execute: (args) => Promise<ToolResult>; // 执行函数

  // 可选属性
  permissions?: {         // 权限控制
    canUse?: CanUseToolFn;
  };
  progress?: {           // 进度回调
    onProgress?: (data) => void;
  };
}

🗣️ 人话解释

想象你要设计一个万能遥控器，可以控制电视、空调、音响。Tool.ts 就是定义这个遥控器接口的地方：

  • name：按钮名称（"电视"、"空调"）
  • description：按钮说明（"打开电视"、"调高温度"）
  • parameters：需要按哪个键、按几次
  • execute：按下按钮后实际执行的指令

💡 架构设计核心思想：分层与隔离

整个架构遵循"依赖倒置原则"：

  • QueryEngine 不直接调用具体工具，而是通过 Tool 接口
  • 工具系统 不感知 UI，只返回结果
  • 权限系统 作为中间层，可插拔

这样设计的好处：任何一层都可以独立替换。比如你可以把终端 UI 换成 Web UI，而不用改 QueryEngine 逻辑。

📍 配置加载代码

restored-src/src/utils/config.ts
getGlobalConfig() 函数负责加载和合并配置

🗣️ 人话解释

配置加载就像准备出门前的检查清单。Claude Code 启动时会按优先级读取配置：

  1. 环境变量（最高优先级）— 如 CLAUDE_CODE_COORDINATOR_MODE
  2. 用户级配置 — ~/.claude/settings.json
  3. 项目级配置 — .claude/CLAUDE.md
  4. 默认配置（最低优先级）— 代码中的默认值

💡 为什么这样设计？

这种分层配置设计让你可以：

  • 全局设置个人偏好（用户级配置）
  • 为特定项目覆盖设置（项目级配置）
  • 临时调试时快速切换（环境变量）

⚡ Streaming 响应处理

📍 流式处理代码

restored-src/src/query.ts
query() 函数处理 streaming 响应

🗣️ 人话解释

Claude 的响应是流式的（像打字一样逐字出现），而不是一次性返回完整答案。这是因为：

  1. 用户体验 — 用户不用等待，可以立即看到部分结果
  2. 长内容支持 — 大段代码或长回复不会超时
  3. 工具调用检测 — 实时判断是否触发了工具调用

💡 为什么需要循环？

QueryEngine 使用递归循环而不是单次调用，是因为：

  • 信息不对称 — AI 不知道文件内容，必须先读取
  • 试错迭代 — 搜索没找到 → 换个关键词 → 再搜索
  • 多步骤任务 — 读文件 → 分析 → 修改 → 验证

🔧 Tool 基类设计

📍 基类定义

restored-src/src/Tool.ts
第 1-100 行，Tool 类型定义和核心接口
Tool 接口的核心结构（简化版）
// 1. 工具描述（告诉 AI 这是什么工具）
name: 'Bash'  // 工具名称
description: '执行 shell 命令'  // 功能描述

// 2. 参数定义（告诉 AI 需要什么参数）
parameters: z.object({
  command: z.string(),  // 命令字符串
  description: z.string().optional(),  // 可选的描述
  timeout: z.number().optional(),  // 可选的超时时间
})

// 3. 执行函数（实际执行逻辑）
execute: async (args) => {
  const result = await exec(args.command)
  return {
    content: result.stdout,
    isError: result.exitCode !== 0
  }
}

// 4. 权限控制（决定能否使用）
permissions: {
  canUse: (context) => {
    // 检查是否有权限执行此命令
    return checkPermission(args.command)
  }
}

🗣️ 人话解释

这就像设计一个万能遥控器的接口规范：

  • name — 按钮标签（"电视"、"空调"）
  • description — 按钮说明书（"打开电视机"）
  • parameters — 按钮参数（音量 +10、频道 5）
  • execute — 按下后实际发送的红外信号
  • permissions — 童锁功能（防止小孩乱按）

🛡️ 以 BashTool 为例看工具实现

📍 BashTool 实现文件

restored-src/src/tools/BashTool/BashTool.tsx
约 800 行，最复杂的工具实现
BashTool 的特殊之处
// BashTool 比其他工具多了很多安全模块：

1. bashPermissions.ts    // 权限规则匹配
2. bashSecurity.ts       // AST 语法分析
3. pathValidation.ts     // 路径约束检查
4. shouldUseSandbox.ts   // 沙箱执行判断
5. modeValidation.ts     // 权限模式验证

// 执行流程也比普通工具复杂：
execute(command) {
  // 1. 权限检查（多层）
  const permission = await checkPermission(command)
  if (!permission.allowed) {
    return { isError: true, content: 'Permission denied' }
  }

  // 2. 安全检查（AST 分析）
  const security = await analyzeSecurity(command)
  if (security.dangerous) {
    return { isError: true, content: 'Dangerous command' }
  }

  // 3. 实际执行
  const result = await exec(command)

  // 4. 结果处理
  return formatResult(result)
}

🗣️ 人话解释

BashTool 就像银行的转账操作，比其他操作多了很多安全步骤：

  • 身份验证（权限检查）— 你能转账吗？
  • 风控检测（AST 分析）— 这转账可疑吗？
  • 限额检查（路径验证）— 超出每日限额了吗？
  • 实际转账（执行命令）— 真正执行操作

💡 为什么 BashTool 这么复杂？

Bash 可以执行任意系统命令，是风险最高的工具。它的设计理念是：

  • 默认拒绝 — 不确定的命令先拒绝，再询问用户
  • 多层防护 — 配置规则 + 分类器 + AST，层层把关
  • 可审计 — 所有命令都记录，可追溯
